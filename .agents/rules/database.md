# Rule: Database Standards

## Scope

All database design: schema definitions, table naming, relationships, indexing, migrations, and query patterns. Applies to PostgreSQL by default; principles apply to any relational database.

---

## Standards

A good schema is:
- Readable: table and column names communicate what they contain, not implementation details
- Normalized: data lives in one place, not duplicated across tables
- Indexed appropriately: common access patterns are fast; over-indexing is avoided
- Migration-managed: all schema changes are in version-controlled migration files
- Minimal: tables exist for real domain concepts, not speculative future needs

---

## Required Practices

**Naming:**
- Tables: `snake_case`, plural nouns — `users`, `orders`, `product_variants`
- Columns: `snake_case` — `created_at`, `user_id`, `stripe_customer_id`
- Primary keys: `id` (UUID preferred for external-facing tables, bigserial for internal)
- Foreign keys: `<table_singular>_id` — `user_id`, `order_id`
- Boolean columns: prefix with `is_` or `has_` — `is_active`, `has_verified_email`
- Timestamps: always include `created_at` and `updated_at` on every table

**Data Types:**
- Use `timestamptz` (timestamp with time zone) — never `timestamp` without timezone
- Use `numeric` or `bigint` for currency — never `float` or `double`
- Use `text` for variable-length strings — avoid `varchar(N)` unless N is meaningful
- Use `uuid` for externally exposed IDs generated by the app
- Use `jsonb` for truly flexible/unstructured data — not as a lazy schema substitute

**Relationships:**
- Always define foreign key constraints — do not rely on application logic to maintain referential integrity
- Use `ON DELETE RESTRICT` as default unless cascade behavior is intentional and documented
- Junction tables for many-to-many: `user_roles`, `order_items` — named as `<entity1>_<entity2>`

**Indexing:**
- Index all foreign key columns
- Index columns used in `WHERE` clauses on large tables
- Index columns used in `ORDER BY` on paginated queries
- Do not index columns with very low cardinality (booleans, small enums)
- Use partial indexes for filtered queries: `WHERE deleted_at IS NULL`
- Use `CONCURRENTLY` when adding indexes to production tables

**Migrations:**
- Every schema change goes in a migration file — no manual schema edits
- Migrations must be reversible (include a rollback) where possible
- Never drop columns or tables in the same migration as the code that stops using them — use two-step deployment
- Test migrations on a copy of production data before running in production

---

## Forbidden Patterns

- Storing JSON blobs instead of normalized columns when the fields are known
- Using `float` for money values
- Storing arrays of IDs in a column instead of a junction table
- Using `timestamp` without timezone
- Creating columns named `data`, `info`, `extra`, or `misc`
- Running ad-hoc schema changes on production databases outside of migrations
- Adding indexes without checking if they will actually be used

---

## Review Checklist

- [ ] All tables have `created_at` and `updated_at` columns
- [ ] All timestamps use `timestamptz`
- [ ] Foreign keys are defined as constraints, not just naming conventions
- [ ] All foreign key columns are indexed
- [ ] No float/double types used for money
- [ ] All schema changes are in migration files
- [ ] No `data` / `info` / `extra` column names
